There is a ''.git'' directory, which has all the git goodness.

|!File or /Dir|!Description|
|/objects|Inside this directory as 2 character directories that house the Git [[objects]].|
|/config|This file has the project-specific configs, overriding global [[.gitconfig]] settings.|
|/refs|This directory has tag information, which eventually provides an object hash.|
|[[HEAD]]|File provides the reference to the current branch.|

The file ''~/.gitconfig'' has global git config information. It is located at $HOME\.gitconfig, which in:
*win8 is ''C:\Users\<user>\.gitconfig''
*linux is ''\home\<user>\.gitconfig''
*linux can also have something at ''\etc\gitconfig''
To list out the global settings on the command line:
*''git config -&dash;global -&#45;list''
*''cat ~/.gitconfig''  (This gives a nice formatting!)
To edit you can: 
*''git config &#45;&#45;global &#45;&#45;edit''
*''vi ~/.gitconfig''
!!Other configs
There are three: global, system, and local. I don't have a system one. You can see if there is a system one: ''git config -&#45;system -&#45;list''
Here they are:<html><pre>
$ git config --list   # This consolidates all configs
$ git config --global -l
$ git config --system -l
$ git config --local -l
#Other options 
$ git config user.name   # Just list user.name
</pre></html>
If you get duplicates, its because more than 1 config files is used, like when you have a local repo you are working with. In that case, the local config supercedes the other ones. For example user.name can appear twice, once as a global config and also as a local config.
Example, running ''"git config -&dash;local -l"'': (or directly: ''cat .git/config'' to get nicely formatted response like below)<html><pre>[user]
	name = Fred Flintstone
	email = fred@rtfm.ca
[remote "origin"]
	url = https://github.com/blah/testconfig.git
	fetch = +refs/heads/*:refs/remotes/origin/*
[branch "master"]
	remote = origin
	merge = refs/heads/master
[alias]
	st = status
	br = branch
	hist = log --pretty=format:'%h %ad | %s%d [%an]' --graph --date=short
</pre></html> 
There are three ways to ignore files in git:
!!First: .gitignore
Use ''.gitignore'' to prevent git from tracking certain files. This file should be in the root directory of a project. Here is an example:
<html><pre>
junk.txt   # Specific file
*.log      # File pattern
secret/    # Directory
</pre></html>
!!Second: global gitignore
You can also create a global .gitignore file, which is a list of rules for ignoring files in every Git repositories on your computer. There are 2 steps to this:
#Create file at ''~/.gitignore_global'' and add some rules to it.
#Let git know about this file, code below...
<html><pre>
git config &#45;&#45;global core.excludesfile ~/.gitignore_global
</pre></html>
!!Third: Explicit repository excludes 
If you don't want to create a .gitignore file in your project to share with others, you can create rules that some files are not committed to the repository. You can use this technique for locally-generated files that you don't expect other users to generate, such as files created by an editor.
Open the file called ''.git/info/exclude'' within the root of your project. 
Any rule you add here will not be checked in, and will only ignore files for your local repository.
Also see [[ignore tracked files]].

#do an [[init]]

Not in this case, but in a case where you are working off of a common repo, you'll want to pull the changes before you push your changes back. 
Issue:  ''git pull origin master''.

There are many websites that help with maintaining a git repo. ~GitLab and GitHub are two such sites, there are others.
Consider these as "servers" and your local machine as the "client".
There are two ways to access the server - a desktop or the terminal. 
You'll need to log into the server and push and pull code.

If you have multiple Github identities (work vs personal) then an important thing is under what User does Github commit under?
When we [[setup]] Git, we typically specify a ''user.name'' and a ''user.email'' in the global [[.gitconfig]] file.
Git will first use a local config setting first, if it finds none, then it will default use to the global settings ([[.gitconfig]]).
The ''user.email'' setting is used to ''determine'' under what Github account to commit as, *if* you have multiple accounts. You have to set local ones each time you clone a repo, or create a repo. 
One trick is to set the global user.email "(empty)" - so when you push, git will choke and ask you to enter user info, name & email. The committer for Github is based on the email.
<html><pre>
git config user.email --global "(nothinghere)"   # To force git to ask you when you attempt to push
git config user.email "wahoo@some.net"           # This will set the local config
</pre></html>
If the email provided doesn't match the github email, it will use the ''user.name'' as the committer.
I believe that if there is no local email, and the global email is not one Git hub knows, it just picks one of the multiple emails it has - it may be the right one, or not!

config.macros.upload.action = function(params)
{
		// for missing macro parameter set value from options
		if (!params) params = {};
		var storeUrl = params[0] ? params[0] : config.options.txtUploadStoreUrl;
		var toFilename = params[1] ? params[1] : config.options.txtUploadFilename;
		var backupDir = params[2] ? params[2] : config.options.txtUploadBackupDir;
		var uploadDir = params[3] ? params[3] : config.options.txtUploadDir;
		var username = params[4] ? params[4] : config.options.txtUploadUserName;
		var password = config.options.pasUploadPassword; // for security reason no password as macro parameter	
		// for still missing parameter set default value
		if ((!storeUrl) && (document.location.toString().substr(0,4) == "http")) 
			storeUrl = bidix.dirname(document.location.toString())+'/'+config.macros.upload.defaultStoreScript;
		if (storeUrl.substr(0,4) != "http")
			storeUrl = bidix.dirname(document.location.toString()) +'/'+ storeUrl;
		if (!toFilename)
			toFilename = bidix.basename(window.location.toString());
		if (!toFilename)
			toFilename = config.macros.upload.defaultToFilename;
		if (!uploadDir)
			uploadDir = config.macros.upload.defaultUploadDir;
		if (!backupDir)
			backupDir = config.macros.upload.defaultBackupDir;
		// report error if still missing
		if (!storeUrl) {
			alert(config.macros.upload.messages.noStoreUrl);
			clearMessage();
			return false;

		if (config.macros.upload.authenticateUser && (!username || !password)) {

			clearMessage();
			return false;

};

bidix.upload.httpUpload = function(uploadParams,data,callback,params)
{
	var localCallback = function(status,params,responseText,url,xhr) {
		url = (url.indexOf("nocache=") < 0 ? url : url.substring(0,url.indexOf("nocache=")-1));
		if (xhr.status == 404)
			alert(bidix.upload.messages.storePhpNotFound.format([url]));
		if ((bidix.debugMode) || (responseText.indexOf("Debug mode") >= 0 )) {

				responseText = responseText.substring(responseText.indexOf("\n\n")+2);
		} else if (responseText.charAt(0) != '0') 
			alert(responseText);
		if (responseText.charAt(0) != '0')
			status = null;
		callback(status,params,responseText,url,xhr);
	};
	// do httpUpload
	var boundary = "---------------------------"+"AaB03x";	
	var uploadFormName = "UploadPlugin";
	// compose headers data
	var sheader = "";
	sheader += "--" + boundary + "\r\nContent-disposition: form-data; name=\"";

	sheader += "backupDir="+uploadParams[3] +

		maxLine = -1;

	if (maxLine != 0) 
		this.tiddler.text = this.tiddler.text + text;
	// Trunck to maxLine
	if (maxLine >= 0) {
		var textArray = this.tiddler.text.split('\n');
		if (textArray.length > maxLine + 1)
			textArray.splice(1,textArray.length-1-maxLine);

	this.tiddler.modified = new Date();

A branch is a pointer to a specific [[commit]]. 
There are 2 types of branches: ''local'' and ''remote'' (or better: ''remote-tracking'').
*''refs/remotes/origin/master'' - this has the ~SHA1 for the latest commit remotely.
*''refs/heads/master'' - this has the ~SHA1 for the latest commit locally.
*''refs/heads/biteme'' - this has the ~SHA1 for the latest commit locally, the branch name is ''biteme''.
In Git a branch should be part of the everyday workflow. 
A branch is used to isolate a development task from the main branch. This can be a major revision all the way down to a small bug fix. 
To work on a [[remote branch locally]] or more likely a [[branch example]].
When you create the branch, let's say called "develop", then at that moment there are two pointers pointing to the same [[commit]], the [[master]] branch and your new ''develop'' branch.
How does GIT know where you are when you are on a ''master'' or ''develop'' branch. The [[HEAD]] is a special pointer that does that.
@@color(red):Caution@@ - when you make changes in the working directory and move to a different branch, the changes will @@follow@@ you! You need to commit them so the working directory is not ''dirty''.
|!Command|!Description|
|git branch|List the branches the repository has.|
|git branch -a|Lists all branches (ie remote ones too).|
|git branch -av|Lists all branches plus one log message.|
|git branch -vv|Lists all branches where you are tracking remotely.|
|git checkout <branch>|Create a branch, does not perform a [[checkout]].|
|git checkout -b <branch>|Create a branch, then checkout to that branch.|
|git branch -d <branch>|Delete a branch, unless there are unmerged changes.|
|git branch -D <branch>|Forces the delete of a branch.|
|git branch -m <branch>|Rename a branch.|

If you use the [[checkout]] command with the branch tag, it will save a step. The ''checkout'' reassigns the [[HEAD]] pointer.

The ''reset'' command. See also [[reset vs revert]] and [[rebase]].
This command can alter existing history.
It should be used mostly to reset a single file ie [[undo staged changes]], which is moving ''some-file'' from [[staging]] directory back to [[working directory]].
* ''git reset <file>'' - Unstage file from staging, place in working. Changes still exist in file.
**Resetting a file in working does nothing.
*''git reset'' - Unstage *all* files in staging, place in working. Changes still exist in all files. 
**Back at prior commit.
*''git reset <file> -&dash;soft <commit>'' - Files still [[staged|staging]], but our HEAD points to commit ''<commit>''. 
**All files changed in between all commits are still there! They are ''not staged'', but "changed" in [[working directory]].
*[[git reset <commit>]] - To reset HEAD back to commit <commit>.
**Unstage *all* files in staging, place in working. Changes still exist in all files, all other files remain available.
**History is lost.
*''git reset -&dash;mixed <commit>'' - This is the default, same as issuing ''git reset <commit>''
*[[git reset --hard]] - Remove *all* staged files. Deleted. Gonzo.
**Back at prior commit. Unstaged files remain.
**History is lost.
*[[git reset --hard <commit>]] - To reset HEAD back to commit <commit>,  *all* staged files are deleted. 
**Back at commit <commit>. Unstaged files remain.
**History is lost.

[[reset]] - this actually deletes a bad commit, actually erasing history.
[[revert]] - undoes a single commit, it doesn't really revert back (delete) to a prior commit, it actually creates a new commit and keeps the history.

SSH is a protocol allowing secure remote login to a computer on a network using public-key cryptography. 
You may need [[ssh setup]].
To create a public & private key, use: @@ssh-keygen@@
To provide keys to remote servers use: @@ssh-agent@@
To add keys to the above sshagent use: @@ssh-add@@
To see if already have some keys: @@ls ~/.ssh@@
Typically a system has just one key, there are two files:
*id_rsa.pub - this is like a lock, anyone can use to lock something & send it to you.
*id_rsa - this is the key, only you have it. it can unlock the above. Can have a password tired to it.
SSH typically looks for the private key in a file in the home directory (''~/.ssh/id_rsa '').  This is stored in an encrypted form, so we have to enter the passphrase which is used to get the encryption key when we are accessing a encrypted resource. To avoid typing it in all the time, we can use [[ssh-agent]], which for the duration of a login session, keeps the unencrypted key in memory.

We use the ''stash'' command to stash changes that we have made so that we can work on a higher priority task. It only stashes files it is [[tracking]]. If you want to stash untracked files use: ''git stash -u''. 
Multiple stashes? Yes, see [[stash multiple]].
You can also stash into a [[branch]]: ''git stash branch new-branch''
The scenario is:

#Make changes and commit, almost done...

Git is a version control system. When you make a commit, Git stores a snapshot of the current state of the system. If a file has not changed, then a link to the previous committed version is done.
All commits are done with an SHA-1 hash, this makes sure nothing is corrupted. 
When you commit, there will be a commit object that points to a "tree" that points to the files that you have commited.
The default name (technically a [[branch]] is [[master]].
The working directory is:
*This is where you work, it is your root directory.
*It has files that are *not* handled by git - they are ''untracked''.
*This is where files are changed, like a scratch pad.
*When you git [[add]] a file, the file is flagged as [[staging]].
*files here will change when you switch branches
*@@uncommited files will remain even when you switch branches@@. It is after all a working directory.
*"git status" shows you what is here
*commited files will be placed on their branch
See [[file example]] or [[staging]].
